SECCIÓN 0 — FIX DE CARGA DE PARTIDOS (antes que nada)
===========================================
El síntoma reportado ("a veces no carga el partido, a veces tarda mucho") en una integración Phaser+React casi siempre tiene UNA de estas causas — revisá y arreglá TODAS:

a) DOBLE MONTAJE POR REACT STRICTMODE: en desarrollo, React 18 con StrictMode ejecuta los efectos de montaje DOS VECES. Si el juego Phaser se crea dentro de un useEffect sin guard, se crean DOS instancias de Phaser.Game compitiendo por el mismo canvas — esto explica perfectamente por qué a veces "no carga" (las dos instancias pelean por el DOM) y a veces "tarda" (una se destruye a medias mientras la otra intenta iniciar). FIX obligatorio:

useEffect(() => {
  let game: Phaser.Game | null = null;
  let destroyed = false;

  if (!gameInstanceRef.current) {
    game = new Phaser.Game(config);
    gameInstanceRef.current = game;
  }

  return () => {
    destroyed = true;
    if (gameInstanceRef.current) {
      gameInstanceRef.current.destroy(true);
      gameInstanceRef.current = null;
    }
  };
}, []); // array de dependencias VACÍO — nunca recrear por cambios de props

b) DATOS DE PARTIDO NO LISTOS ANTES DE INICIAR LA ESCENA: si matchData (equipos, jugadores, stats) se lee de forma async (fetch, estado de carrera) pero la escena de Phaser arranca antes de que esos datos estén disponibles, la escena puede fallar silenciosamente o quedar en loop de espera. FIX: agregar un estado de loading explícito en el componente React que NO renderiza el contenedor de Phaser hasta que matchData esté 100% listo, con un timeout de seguridad (ej: si pasan 8 segundos sin datos, mostrar error explícito en vez de spinner infinito).

c) MÚLTIPLES INSTANCIAS DE PHASER.GAME ACUMULADAS: si el usuario navega entre pantallas (menú → partido → menú → partido) varias veces sin que la instancia anterior se destruya correctamente (game.destroy(true) con el parámetro true para remover también el canvas del DOM), cada vez queda más lento por instancias fantasma consumiendo recursos. Confirmá que TODA navegación fuera de la pantalla de partido dispara la destrucción completa de la instancia anterior.

d) CONFIG DE PHASER MAL DIMENSIONADA: si el div contenedor (match-canvas-root) no tiene un tamaño definido en el momento en que Phaser lee sus dimensiones (por ejemplo si el CSS todavía no aplicó layout), Phaser puede inicializar con dimensiones 0x0 y quedar "colgado". FIX: usar Phaser.Scale.RESIZE con el contenedor ya montado y medido (getBoundingClientRect) antes de instanciar el juego, no en el mismo tick de render.

Implementá un log de consola temporal en cada etapa (mount, datos listos, Phaser.Game creado, primera escena iniciada) para que puedas confirmar en qué paso específico se traba cuando "tarda mucho" o "no carga".

SECCIÓN 1 — FÍSICA HAXBALL (valores reales, recalibrados MÁS LENTO)
===========================================
Haxball corre a 60Hz con estos valores documentados por la comunidad (stadium "Classic"):
- Jugador: radio 15, bCoef 0.5, invMass 0.5 (masa 2), damping 0.96/tick (MULTIPLICATIVO, no lineal), acceleration 0.1/tick
- Pelota: radio 10, bCoef 0.5, damping 0.99/tick, invMass 1 (masa 1)
- Ratio de masa jugador:pelota = 2:1

IMPORTANTE — CALIBRACIÓN DE VELOCIDAD PARA ESTE PROYECTO: las versiones anteriores quedaron demasiado rápidas y la jugabilidad se sintió mal por eso. Esta vez calibrá para una velocidad NOTABLEMENTE MÁS LENTA Y CONTROLABLE:

export const PLAYER_MAX_VEL_TARGET = 118; // px/s en equilibrio — lento y controlable, NO subir este número
export const PLAYER_DAMPING = 0.96;       // directo de Haxball, no cambiar
export const BALL_DAMPING   = 0.985;      // pelota rueda con fricción moderada, ni eterna ni corta

Con damping 0.96/tick a 60Hz, la velocidad de equilibrio es accel/(1-damping) = accel*25. Para lograr 118 px/s: accel_per_tick ≈ 118/25/60 ≈ 0.0787 (ajustar empíricamente ±15%, PERO el objetivo final debe sentirse claramente más lento y "pesado" que las versiones anteriores, priorizá control sobre velocidad).

IMPLEMENTACIÓN — fixed timestep obligatorio (desacopla física de FPS del dispositivo):

private physicsAccumulator = 0;
private readonly FIXED_STEP = 1000 / 60;

update(time: number, delta: number) {
  this.physicsAccumulator += delta;
  while (this.physicsAccumulator >= this.FIXED_STEP) {
    this.fixedPhysicsUpdate(this.FIXED_STEP / 1000);
    this.physicsAccumulator -= this.FIXED_STEP;
  }
  this.syncVisuals();
}

Dentro de fixedPhysicsUpdate(), aplicar el damping MULTIPLICATIVO manualmente (no usar setDrag() lineal de Arcade):

player.body.velocity.x *= PLAYER_DAMPING;
player.body.velocity.y *= PLAYER_DAMPING;
player.body.velocity.x += inputVector.x * PLAYER_ACCEL_PER_TICK;
player.body.velocity.y += inputVector.y * PLAYER_ACCEL_PER_TICK;

ball.body.velocity.x *= BALL_DAMPING;
ball.body.velocity.y *= BALL_DAMPING;

MASAS: player mass = 2.2, ball mass = 1.
REBOTES: player-vs-player bounce ~0.08 (suave), ball vs paredes/postes bounce ~0.6 (vivo pero no exagerado).

Los jugadores se representan SIEMPRE como:

PROHIBIDO usar Phaser.GameObjects.Arc sueltos para los jugadores — en Phaser 3.90 pierden su render tracking cuando se agrega una segunda cámara (minimapa) después de crearlos, causando que desaparezcan tras el primer frame. El Container es inmune a esto porque se renderiza como unidad atómica.
